01 · Overview

What tanker is

tanker is a structured set of Claude Code skills, agents, and rules. It replaces ad-hoc prompting ("please write me X") with a repeatable pipeline where each phase of a build has a defined skill, a quality gate, and an explicit handoff to the next step.

The core premise: Claude is capable, but without structure every session starts from scratch. tanker gives Claude a role for every phase — product thinking, architecture, planning, execution, review, ship, monitor — plus a persistent project brain that remembers conventions, decisions, and landmines across sessions.

If there's a skill for the phase you're in, use it. Skipping to raw code is how you get AI slop.

tanker now also runs in autopilot — one command, brief to live product. See the next section. The manual phase-by-phase flow still exists for cases where you want hands-on control.

The whole system, reduced to what you actually type and read. No jargon.

The one rule: when tanker stops at a gate and asks for review, actually review it. The whole system depends on you being the head of product at those two moments. Everything else is automated so you have the attention to spend there.

02 · Autopilot

Autopilot mode — /cto

tanker had all the building blocks (32 skills, 4 review agents, three-layer memory) but no top-level orchestrator that strung them together end-to-end. /cto closes that gap. It is the autonomous CTO layer — you give it a one-line product brief, it produces a deployed product.

one command

/cto "AI business analyst for Indian D2C sellers — connects 6 SaaS tools, chat with your data"

What it does, end to end

brief in
  └── intake → state.json created (outputs/<slug>/state.json)
      └── context load: brain.md + memory + Obsidian vault search (parallel)
          └── github-scout agent → ≥5 reference repos, what to copy, what to skip
              └── /grill → /benchmark?
              └── /prd  → exhaustive PRD + HTML wireframes per screen + landing
                  └── prd-reviewer agent → PASS / PASS WITH FIXES / BLOCK
                      └── BLOCK → /prd retry (max 2)
                  └── 🛑 HARD STOP — Human review gate 1
                      └── owner reads outputs/<slug>/prd/index.html
                      └── owner replies "approved" / "fix: ..." / "abort"
              └── /architect → /createplan → /advisor
                  └── PROVISION (parallel)
                      ├── gh-provisioner       repo + branch protection + secrets
                      ├── supabase-provisioner project + RLS schema (Management API)
                      ├── vercel-provisioner   project linked to GH + env vars
                      └── railway-provisioner  service + healthcheck (if backend)
                  └── BUILD (parallel engineering pool)
                      ├── frontend-engineer  → /design-shotgun → code → PR
                      ├── backend-engineer   → /backend-builder → PR
                      ├── data-engineer      → versioned migrations → PR
                      └── content-engineer   → copy + SEO + README → PR
                  └── per PR: pre-merge agent + /autoresearch-review
                      └── on PASS: gh pr merge --squash
                      └── on FAIL: dispatch the engineer again with the report (max 2 retries)
                  └── spin up LOCAL preview (npm run dev / uvicorn / etc)
                      └── mvp-reviewer agent → PASS / PASS WITH FIXES / BLOCK
                          └── BLOCK → re-dispatch build subagents (max 2)
                      └── 🛑 HARD STOP — Human review gate 2
                          └── owner opens http://localhost:<port>
                          └── owner replies "approved" / "fix: ..." / "abort"
                  └── /deploy production → smoke test → live
                      └── healthcheck fails → auto-rollback
                  └── /monitor (Sentry + Plausible + uptime)
                  └── final report: prod URL, repo URL, dashboards

Human-in-the-loop gates

The owner is the head of product in this loop. Autopilot's job is to never bring the owner half-finished work — review agents pre-qualify both gates so manual review is always against complete, working artifacts.

GatePre-check (auto)Owner reviewsApprove action
Gate 1 — PRD prd-reviewer agent: every feature → screen, every screen → all states, no AI slop in mocks, ≥3 open questions, no hand-wave language outputs/<slug>/prd/index.html + prd.md Reply approved → /cto proceeds to /architect; reply fix: ... → re-run /prd with feedback
Gate 2 — Local MVP mvp-reviewer agent: every screen 200s, golden paths walk, all states render, responsive at 375/768/1440px, no console errors Live local URL: http://localhost:<port> Reply approved → /cto deploys to production; reply fix: ... → re-dispatches build subagents
Why two gates and not more: the PRD gate catches strategic mistakes (wrong product) when fixing them is cheap. The MVP gate catches build mistakes (wrong UX, missed features) before any production traffic. Anything else (review every PR, review every screen as it's built) drowns the owner in noise. Two gates, both pre-qualified, is the right ratio.

Why autopilot is production-grade (not just throwaway)

The safety isn't "stop and ask at every step" — it's that every step is reversible and observed. The rails are architectural:

  • Branch protection on main — provisioner sets it on day one. Direct pushes blocked. Every change goes through a PR.
  • Versioned migrations only — schema changes live in supabase/migrations/<ts>_*.sql. Reversible by design.
  • Mandatory preview deploys — every PR gets a unique Vercel preview URL before it can touch prod.
  • Healthcheck-gated rollback — Railway and Vercel auto-revert on a failed /health response.
  • State.json checkpointed every phase — resume any session with /cto --resume <slug>. Never lose work to a session timeout or a failed step.
  • pre-merge agent before any merge — autoresearch + review run before main is touched. No "let's hope it's fine."

Resume + status

CommandEffect
/cto "<brief>"Fresh run. Default mode is autopilot.
/cto "<brief>" --gatedSame flow, but pauses at each phase boundary for human approval.
/cto --resume <slug>Reads state.json, skips done phases, continues from the next.
/cto --status <slug>Read-only summary — phase, infra URLs, PR statuses, errors.

Failure handling

If this happens…autopilot does this
Vault key missingSTOP. Prompt to /vault-add <service> and resume.
Provisioner API failureSTOP. Surface the API error verbatim. No blind retry.
Build subagent produces broken codepre-merge catches it; subagent is re-dispatched with the failure report. Max 2 retries.
Deploy healthcheck failsAuto-rollback to last green deploy. Surface logs.
/advisor flags CRITICAL issueSTOP after planning phase. Owner reviews — do not auto-apply cross-model edits over author-model output.
Pre-flight before first run: register credentials with /vault-add github, /vault-add vercel, /vault-add supabase, /vault-add anthropic (and railway / sentry / plausible if needed). Provisioners read from the vault — never inline tokens, never paste them in chat.

02b · The pattern

The owner's loop — three moments

You only do three things across the entire build. Everything else is automated.

Moment 1 — Trigger

One sentence. Type /cto "<your idea>". Walk away. Autopilot loads context, scans references, drafts a PRD, and a review agent sanity-checks it.

Time: 30 seconds.

Moment 2 — Review the PRD

You see it only after it passed the prd-reviewer agent. Open outputs/<slug>/prd/index.html — clickable wireframes for every screen, every state, plus the narrative PRD. Reply approved or fix: <feedback>.

Time: 15–20 minutes of careful reading. You decide what gets built.

Moment 3 — Review the MVP

You see it only after it passed the mvp-reviewer agent. Open http://localhost:<port>. Real working build. Click through every flow. Reply approved or fix: <feedback>. On approval, autopilot ships to production.

Time: 10–15 minutes of clicking. You're the final QA.

~30 minutes of head-of-product attention. Live product at the end. Reviewable artifacts at every stage.

02c · Comparison

Manual workflow vs autopilot

Both still exist. Use whichever fits the situation. The difference is who drives.

AspectManual flowAutopilot (/cto)
Commands you type 10–14 slash commands, in order, one at a time 1 command + 2 review responses
Owner attention 4–8 hours of active driving + context-switching ~30 minutes of focused product review
Infra setup Manual: create GitHub repo, Supabase project, Vercel project, Railway service, push secrets, configure each Automatic: 4 provisioner agents in parallel via vault credentials
PRD format /architect output (technical) — no exhaustive screen list, no wireframes /prd output: 12 mandatory sections, every feature → screen, every screen → all states, HTML wireframes per screen + landing
Quality gates You remember to run /autoresearch-review before merging. pre-merge agent before PR. Built into the loop — pre-merge on every PR, prd-reviewer + mvp-reviewer auto-pre-qualify human gates
Resumability Reconstruct via /context-restore — re-read context.md + git log state.json checkpointed every phase. /cto --resume <slug> picks up exactly where it stopped.
Best for Existing codebase work, surgical fixes, when you want full control of every decision New products, greenfield builds, when the goal is "live product by end of day with my taste applied at the right gates"
Both flows share the same primitives: the 32 skills, the 4 review agents, the 3-layer memory, the always-on rules. Autopilot is the orchestrator that strings them together — manual is you doing it yourself. Same parts, different driver.

03 · System

The vault — credentials for autopilot

For autopilot to provision real infrastructure, it needs credentials. tanker stores them locally at ~/.claude/vault/credentials.json with strict permissions and never logs the values.

PropertyValue
Location~/.claude/vault/credentials.json (outside any project repo)
PermissionsDirectory 0700, file 0600 — owner read/write only
FormatJSON, namespaced by service: .github.pat, .vercel.token, .supabase.access_token…
Per-project keyse.g. .supabase.<slug>.url — each project gets its own Supabase keys when provisioned
Read byProvisioner agents only, via jq. Never echoed to chat, never put in commits.

Adding a credential

Use the /vault-add skill. It prompts you to open the file directly in your editor or accept input through a hidden shell prompt — the value never enters the chat transcript.

/vault-add github
# → directs you to github.com/settings/tokens (scopes: repo, workflow, delete_repo)
# → confirms the key was written without ever printing the value
# → optionally runs `gh auth status` to verify the token works

Supported services

ServiceRequired keysUsed by
githubpat, usergh-provisioner
verceltoken, team_id (optional)vercel-provisioner
railwaytokenrailway-provisioner
supabaseaccess_token, org_idsupabase-provisioner
cloudflareapi_token, account_id(future) cloudflare-provisioner

If a credential ever leaks: rotate at the provider first, then update vault, then audit recent agent outputs. Never assume "no one will see the log" — rotate.

03 · Flow

Feature on existing product

Skip the full pipeline. You already have architecture and conventions locked in the project brain.

feature chain

/createplan→ /execute→ /autoresearch-review→ /ship

If the feature is large enough that you're unsure about design, insert /architect before /createplan. If UI is involved and you have no reference, /ui-hunt first.

04 · Flow

Bug fix with unknown cause

bug chain

/debug→ /test-gen→ /autoresearch-review→ /ship

/debug traces symptom → root cause using a structured elimination method (no guess-and-check). /test-gen writes a regression test so the bug can't come back silently.

05 · Flow

Static site or landing page

static site chain

/ui-hunt→ /design-shotgun→ /static-site-replicator→ site-eval agent

The site-eval agent is mandatory before declaring a static site done. It audits visual parity, brand consistency, animations, responsiveness, and accessibility — returns PASS / CONDITIONAL / FAIL.

07 · Reference

Agents — when to delegate

Agents are specialized subprocesses. They start with no context, so brief them fully. Use them to (a) parallelize independent work, (b) protect your main context from massive outputs, or (c) hand a single external service to a scoped writer (provisioners).

Read-only review agents

AgentWhen to use
exploreBefore planning any feature or fix — fast codebase exploration, returns structured report
reviewFocused code review on any significant change — reads only, returns severity-tagged findings
site-evalMandatory before declaring any static site done
pre-mergeBefore every PR merge — combines review + autoresearch into one pass

Research + HITL pre-qualifier agents NEW

AgentWhen to use
github-scoutBefore /architect on any new build — searches GitHub for ≥5 prior-art repos, reads READMEs + manifests, returns a Reference Brief covering the convergent stack, patterns to lift, footguns observed, and a first-principles override.
prd-reviewerPre-qualifies a PRD bundle before owner sees it (Gate 1). Checks every feature → screen mapping, every screen → all states, no AI slop in mocks, no hand-wave language. Returns BLOCK / PASS WITH FIXES / PASS.
mvp-reviewerPre-qualifies a local MVP build before owner sees it (Gate 2). Spins up the local server, walks every screen + golden path, checks responsive, validates all states render, runs accessibility checks. Returns BLOCK / PASS WITH FIXES / PASS.

Provisioner agents (autopilot infra) NEW

Each provisioner reads from the vault, talks to one external service via REST or GraphQL (no CLI dependencies), and returns JSON metadata that gets appended to state.json. All idempotent — re-running on existing infra is safe.

AgentServiceWhat it does
gh-provisionerGitHubCreates repo, sets branch protection on main, pushes secrets from vault, scaffolds .gitignore + a placeholder CI workflow
supabase-provisionerSupabaseCreates project via Management API, scaffolds versioned migrations + RLS, captures keys to vault under per-project namespace, pushes to GitHub repo secrets
vercel-provisionerVercelCreates project linked to GitHub, pushes env vars from vault, configures custom domain, triggers preview deploys on every PR
railway-provisionerRailwayCreates project + service, attaches Postgres if needed, configures healthcheck on /health, sets restart policy ON_FAILURE/3
Why the four provisioners are separate, not one big "deploy everything" agent: each platform's API has different auth, error codes, and idempotency semantics. Keeping them isolated means a Supabase outage doesn't block Vercel, and you can swap Railway for Fly.io later without touching the others.

08 · System

The three-layer memory system

Claude sessions are stateless by default. tanker solves this with three files that auto-load at the start of every session:

FileWhat it holdsUpdated by
.claude/brain.mdConventions, decisions, pitfalls, preferences — cumulative, stable/learn after any meaningful session
.claude/context.mdWhere the last session left off; next steps; open threads/context-save before ending a session
.claude/git-state.mdCurrent branch, last commit, working tree statusStop hook — every turn, automatically
CLAUDE.mdMirror of the above so it loads in every session via Claude Code's auto-load/remember after /learn
outputs/<slug>/state.jsonLive state of an autopilot run — phase, infra IDs, PR statuses, errors. Enables resume./cto appends every phase
~/.claude/vault/credentials.jsonAPI tokens for autopilot infra. Outside the project, 0600 perms./vault-add
Rule of thumb: run /learn at the end of any session where something important was decided. Run /remember right after. Run /context-save if work is unfinished. That's the entire maintenance loop.

09 · System

Skill chaining — how skills hand off

Every skill ends with a line pointing at the next skill. Two styles of handoff:

Automatic

/ui-hunt → /design-shotgun (when you say yes)

/execute step N → step N+1

/autoresearch-review PASS → prompts /ship

Prompt-and-confirm

/grill → "Ready for /architect?"

/design-shotgun direction chosen → "Proceed to /execute?"

/ship opens PR → "Run pre-merge agent?"

Handoffs are explicit prompts, not footnotes. You should never have to guess what comes next.

11 · Guardrails

Anti-patterns — the AI slop fingerprints

These are banned in tanker. The builder-ethos rule enforces them, site-eval fails on them:

  • Purple / violet / indigo gradients as the primary palette (#6366f1–#8b5cf6 range)
  • 3-column feature grid with icon-in-circle + bold title + 2-line description
  • text-align: center on more than 60% of text containers
  • Uniform bubbly border-radius (>80% of elements using 16px+)
  • Generic hero copy: "Welcome to X", "Unlock the power of…", "Streamline your workflow"
  • More than 3 font families on a page
  • Heading levels that skip (h1 → h3)
  • Missing hover / focus states on interactive elements
  • outline: none without a visible focus replacement
