## Conventions
*How this project does things: naming, file structure, patterns, tech choices.*

- Skills live at `.claude/skills/[name]/SKILL.md` with YAML frontmatter (`name`, `description`, `triggers`, `args`), followed by phases, output format, and rules sections
- Always-on rules live in `.claude/rules/` (builder-ethos, code-standards, static-site-standards)
- Agents live in `.claude/agents/[name].md`
- All skill output goes to `outputs/[project-name]/`
- Every skill ends with a handoff line pointing to the next skill in the chain
- Autopilot is the orchestrator layer — `/cto` runs the manual chain end-to-end, dispatching skills + provisioner subagents in parallel
- Provisioner agents (gh, supabase, vercel, railway) hit each platform's REST/GraphQL API directly — no CLI dependencies. One agent per service.
- All credentials live at `~/.claude/vault/credentials.json` (0700 dir, 0600 file). Never logged, never echoed, never in commits. Provisioners read with `jq`.
- Per-project state at `outputs/<slug>/state.json`. Every `/cto` phase appends to `phases_done` so any session can `/cto --resume <slug>`.
- Local semantic retrieval over the Obsidian vault via ChromaDB at `~/.claude/brain-index/data/`. All embeddings local (chromadb default model). Two collections: `brain` (vault) + `refs` (curated GitHub repos). Manifest at `~/.claude/references/repos.yaml`.
- `/cto` Phase 1 (context load) calls `python query.py "<keywords>" --collection brain` and `--collection refs`. github-scout Phase 0 consults curated refs as Tier 0 priority before any GitHub-wide search.
- **CEO-facing PDFs use Anthropic editorial CSS** mirroring `~/Documents/SDN/SDN_Final_v3.html` — Tiempos Headline (display) + Source Serif 4 (body) + Inter (UI) + JetBrains Mono (code), cream `#faf9f6` canvas, paper-white centered 880px column, clay accent `#cc785c`, deep navy headlines `#181818`. Print CSS with `@media print { section { break-before: page; } }`. Render via `/Applications/Google Chrome.app/Contents/MacOS/Google Chrome --headless --disable-gpu --print-to-pdf=X --no-pdf-header-footer --virtual-time-budget=10000 file://...`.
- **Money-card visuals beat prose tables for stakeholder leak/loss sections.** Big rupee figure (44px Tiempos), small uppercase eyebrow, agent name in clay accent, 2-column grid. Saahil-class readers latch onto the number not the prose.
- **Per-agent specs MUST include**: trigger (cron expression OR webhook topic + FastAPI route), data sources (specific `_u` tables + columns), exact write action (connector + endpoint + template id), action tier (T1/T2/T3), target ₹ outcome (derived inline, not asserted).

## Decisions
*Why key choices were made. Prevents re-litigating settled questions.*

- **Kit name is tanker:** Renamed from `my-builder-kit` / `claude-builder-kit` to `tanker` — *Reason: personal naming convention, repo is `tanishg98/tanker`*
- **Three-layer memory system:** Global `~/.claude/CLAUDE.md` rule auto-reads `.claude/brain.md` + `.claude/context.md` + `.claude/git-state.md` at every session start — *Reason: CLI sessions start blank; CLAUDE.md is the only thing that auto-loads*
- **Stop hook for git-state:** `~/.claude/settings.json` Stop hook runs `sync-memory.sh` async on every turn, writes `.claude/git-state.md` — *Reason: always-current branch/commit without manual /context-save*
- **`/remember inject` is semi-manual:** Injects brain into CLAUDE.md for machines without the global rule — *Reason: requires Claude to summarize brain.md, can't be done in a shell script*
- **Skill chaining via SKILL.md instructions:** Skills end with explicit handoff lines; Claude follows them using the Skill tool — *Reason: no native platform chaining mechanism; instruction-following is the only approach*
- **`/ui-hunt` is tanker's unique differentiator:** Finds best-in-class products before building so designs have a real reference — *Reason: root cause of AI slop is building without a reference*
- **Autopilot defaults to HITL, not full-auto:** `/cto` STOPs at two human-review gates (post-PRD, post-local-MVP) by default. Owner is the head of product. `--full-auto` exists as an explicit opt-out. *Reason: agents pre-qualify the work, owner reviews complete artifacts at the right moments — full-auto is reserved for throwaway prototypes.*
- **Production-grade autopilot rails are architectural, not behavioral:** branch protection on main (gh-provisioner), versioned migrations only (supabase-provisioner), mandatory preview deploys (vercel-provisioner), healthcheck rollback (railway/vercel), state.json checkpointing every phase. *Reason: "stop and ask" is brittle; reversible-by-design is robust.*
- **Local embeddings, not API:** brain-index uses chromadb default model (`all-MiniLM-L6-v2`, ~80MB CPU-only). *Reason: vault is private; embedding it through any external API contradicts the privacy posture of the vault itself.*
- **Curated refs > generic GitHub search:** `/cto-add-ref` lets owner curate repos worth pulling patterns from. github-scout consults this Tier 0 before wider search. *Reason: training-time GitHub priors are stale; owner's curated taste is current and personal.*
- **Connector framework — `Connector` ABC pattern:** Every external SaaS source subclasses one ABC with `start_oauth / complete_oauth / connect_with_credentials / test_connection / sync_stream`, returns `NormalizedRecord(target_table, unique_keys, data)`. New connectors are mapping exercises, never framework rewrites. Streams pull-loop with cursor; framework handles upsert by `unique_keys`. *Reason: 6+ connectors that cleanly slot in proves the abstraction; future connectors (Amazon SP-API, Klaviyo, Tally) plug in without touching `sync.py`.*
- **Universal `_u` tables, not source-shaped tables:** `order_u`, `payment_u`, `shipment_u`, `ad_spend_u`, `message_u` etc with `source` + `source_id` + `raw jsonb` provenance. Chat tools query ONE schema. *Reason: source-shaped tables (`shopify_orders`, `amazon_orders`) force chat tools to know every connector's shape — explodes complexity at N connectors.*
- **`raw jsonb` on every row + `get_raw` chat tool:** When the chat needs a field we didn't normalize, model calls `get_raw(table, row_id)`. Eliminates "add a column to capture field X" migrations. *Reason: schema migrations are the #1 reason connector platforms slow down at year 2 — sidestep entirely.*
- **Tenant isolation = app-code RLS-binding + Postgres FORCE RLS as second line:** SUPERUSER bypasses RLS unconditionally — even with FORCE — so backend MUST `SET LOCAL ROLE app_rw` (non-superuser) per-transaction in addition to `set_config('rls.merchant_id', ...)`. Chat's `run_sql` switches to `app_ro_chat` (read-only). *Reason: Supabase pooler logs in as `postgres` superuser by default; we'd silently leak everything without the role switch.*
- **Cove palette (Triple Whale-inspired light) > Bourbon Ink (Linear-grade dark) for SMB tools:** Light mode with #FCFCFD canvas, #2563EB brand-blue, generous whitespace, soft shadows, Geist Sans. *Reason: Indian D2C operators are not Linear power-users — they want Stripe/Triple Whale familiarity. Dark mode reads as "engineering tool" to them. Style follows the user, not the builder.*
- **OAuth where possible, guided key-paste otherwise, concierge as the safety net:** Shopify / Razorpay / Meta / Google → OAuth (one click). Shiprocket / Interakt / AiSensy → key paste (no OAuth offered). Every modal carries "Don't have your keys? Book a 10-min call" as the final fallback for non-technical users. *Reason: V1 design partners onboarded by concierge call doesn't need polished key-paste UX; ship with the floor instead of perfecting the ceiling.*
- **Anon Supabase keys: legacy JWT format = anon-JWT signed with project's HS256 secret.** You can mint a working anon key from the JWT secret in dev. But the API gateway only accepts the actual published anon key from the dashboard for new projects. *Reason: discovered when minted-key broke at the gateway; fall back to dashboard-paste, document this for future Supabase setups.*
- **Supabase free-tier direct DB host is IPv6-only.** Connection from IPv4-only networks 503s. Use the **Session Pooler** URL (port 5432, host `aws-0-<region>.pooler.supabase.com`, user `postgres.<project-ref>`) — IPv4-routable, supports SQLAlchemy prepared statements. *Reason: this single quirk cost ~30 minutes; codify it in any new Supabase setup checklist.*
- **Shiprocket `.in` TLD vs `.co` TLD:** `apiv2.shiprocket.co` goes through CloudFront which 503s anti-bot; `apiv2.shiprocket.in` hits AWS ELB directly and works. Always default to `.in`. *Reason: empirically verified during D2C OS Shiprocket connector debug.*
- **Public-facing brand = Rocketizer (with z, per rocketizer.ai).** Internal codename = d2c os. Repo, channels, codebase use d2c os; landing page, PDFs, marketing use Rocketizer. Tagline: "a D2C OS". *Reason: Rocketizer is Shiprocket's official AI brand; aligning publicly avoids the Visibl-style internal-vs-external collision.*
- **V1 = chat + 10 ₹-saving agents in 14 days, not chat-only.** Day 2 = build the shared `Agent` runtime base class (`trigger() · reason() · act()` hooks + tool-use loop over `_u` schema + Policy gate); Days 3–7 = ship 10 thin subclasses (~200 LoC each); Days 8–14 = polish + onboard + launch. *Reason: shipping chat-only first cedes the ground to Kwik AI; agents on top of the connector layer is the wedge.*
- **Floor scenario for the 14-day sprint** = 5 agents (RTO Saver · Cart Recovery · Weight Disputes · Cash Brief · NDR). If the runtime is right by Day 2 and only 5 agents are live by Day 7, we still launch — these 5 alone produce ₹6–11L/mo per merchant and beat Kwik's chat. *Reason: Sonnet advisor flagged "high-risk timeline with no resolution" — having a documented floor converts the risk from "hope" to "plan".*
- **The 10 V1 agents:** RTO Saver, Cart Recovery, Cash Brief, Ad Pruner, Weight Disputes, NDR Resolution, Inventory Sentinel, WA Support Bot, Review Harvest, Cohort Agent. Aha-velocity-ranked: Day-1 (Cash Brief, RTO Saver, Cart Recovery, Ad Pruner) → Day-2 (WA Support Bot) → Day-5+ (NDR, Weight Disputes, Inventory Sentinel) → Day-10+ (Cohort, Review Harvest). Total ~₹7–14L/mo recovered/saved per ₹50L GMV brand.
- **Pricing ladder for Rocketizer:** Beta (free, V1 closed) → Solo ₹999 → Growth ₹2,999 → Scale ₹9,999 → **Atlas ₹49,999+** (sales-led, managed deployment, dedicated CSM, SLA). **Custom Build Studio** = consulting tier on top of Atlas — bespoke agents at ₹5–25L per build, 2–4 wk delivery, 70-80% gross margin. *Reason: aligns under Rocketizer.ai's existing Launch/Growth/Atlas tiers; SMB self-serve below ₹49K, enterprise managed above.*
- **Shiprocket platform read-mechanism (locked):** prefer Shiprocket-internal API gateway with read-only service token; fallback = scheduled pull from BigQuery exports. Decision locked Day 1. *Reason: direct DB replica access is faster but blocks if SR platform team can't grant; gateway is universally available.*
- **AUTH_MODE=demo for V1.** Single shared merchant per design partner, separate Supabase project per partner for tenant isolation, no per-user auth. Supabase Auth (magic link + Google) wires in V1.5. *Reason: 5 design partners onboarded by hand don't need user auth; building auth in the 14-day sprint is overhead that would consume runtime-build capacity.*
- **Run /advisor (cross-model peer review) before sending any single-model-authored PRD/strategy to Saahil or external stakeholders.** Sonnet reads what Opus authored → catches: undefended ₹ claims, missing pricing/team ask, fabricated TAM headlines, weak risk mitigation, internal-tool-as-current-moat dressing. Apply ALL critical findings; ignore advisor only with explicit owner approval. *Reason: 2026-04-28 advisor run on Strategy v2 caught 6 high-impact gaps; v3 incorporated them all.*
- **Rocketizer architecture is 4 layers, not 3. Agents and chat are SIBLINGS, not stacked.** Connector layer → Data layer (canonical `_u` schema in our own DB) → {Chat layer, Agent runtime} as parallel consumers. Common confusion: drawing agents "on top of" chat. They have different triggers (chat = human-synchronous; agent = cron/webhook-async) but share connectors, the data layer, the tool catalog, the idempotency table, and per-customer rate limits. *Reason: drawing it wrong leads to designing agents as "chatbots that run themselves" — the failure mode of 11x/Artisan, already flagged in CLAUDE.md.*
- **Don't consume third-party MCPs upstream as a substitute for being an app.** Building Rocketizer on top of (e.g.) Shopify's MCP is wrong-shaped: MCP is request/response only (no webhooks → no real-time agent triggers), stateless (no persisted history → chat queries become slow live API calls), single-user-shaped (multi-tenant SaaS does not fit), and exposes only a curated subset of vendor APIs. Rocketizer must be a **proper Shopify/etc. app** with persistent OAuth tokens, own DB, webhooks, normalization layer. *Forward-looking:* publishing a **Rocketizer MCP server** later (so merchants can chat with their Rocketizer data from Claude Desktop / Cursor) is the legitimate use of MCP — that is **downstream distribution**, not **upstream data sourcing**. Two different directions; don't conflate.
- **Rocketizer V1 has 6 production cliffs that demos always pass and prod always fails.** Documented in detail at `~/.claude/projects/-Users-tanishgirotra1-projects-Tanker/memory/project_rocketizer_v1_failure_cliffs.md`: (1) multi-tenancy/RLS, (2) token-refresh storms, (3) idempotency, (4) LLM non-determinism, (5) webhook security/replay, (6) Playwright-at-scale (avoid in V1). Each must be designed in from Day 2 of the runtime build — bug-fix retrofitting any of these in week 3 = a breach, an outage, or a customer-spam incident. Owner reviews engineer code through this lens.
- **Read/write asymmetry in the connector + agent + chat layer is structural, not stylistic.** Read tools = fire instantly, no confirmation, generous rate limits. Write tools = idempotent (deterministic action_id + UNIQUE constraint), human-confirmation required when chat-initiated, structured-output-locked when agent-initiated, audit-logged, share a per-customer rate-limit budget across ALL agents. Tag tools `kind: "read" | "write"` in the catalog; gate writes at the backend. *Reason: treating them symmetrically is the #1 source of "the agent refunded the wrong customer" bugs.*

## Pitfalls
*What broke, what was confusing, what not to repeat.*

- **`advisor` skill ambiguity:** TWO advisor skills now exist. Global `~/.claude/skills/advisor` = Claude API Opus+Sonnet pattern for building API apps. Tanker `.claude/skills/advisor/` = cross-model peer review (e.g. Sonnet reviews Opus's PRD). Always confirm which one the user means. Default in tanker sessions = the peer-review one.
- **Never write a PRD for a competitive product without running `/benchmark` first.** Prose teardowns let features slip — they describe each competitor's top-line pitch, not their full feature set. Always produce the exhaustive feature matrix (rows = atomic features, cols = competitors, cells = ✅/❌/⚠️) BEFORE any PRD work. Every PRD feature must map back to a matrix row with ship/defer/skip + reason. Learned the hard way on Shiprocket Visibl — missed auto-competitor-discovery + auto-prompt-synthesis + per-prompt rank history because the teardown was prose, not matrix.
- **When building a product inside a larger org (Shiprocket, etc.), always ask what internal teams are building in the same space BEFORE drafting features.** Large orgs ship in parallel — assume overlap until proven otherwise. The Visibl internal tool at Shiprocket was invisible to external teardown and nearly caused duplicate effort.
- **Cross-model peer review (`/advisor`) is mandatory on any single-model-authored PRD, plan, or spec** before it goes to a stakeholder. One model's blind spots = another model's obvious flags.
- **Don't push directly to main on the tanker repo.** Sandbox blocks it (correctly — it matches the branch-protection rail we just built). Always work on a feature branch + PR. Local `main` should be force-reset to `origin/main` after committing if you accidentally landed on local main.
- **chromadb install is ~200MB.** First-run brain-index setup downloads the embedding model. Don't auto-install on someone else's machine without explicit consent — trigger only on `/brain-index` invocation.
- **PRD without screen states ≠ a PRD.** prd-reviewer auto-BLOCKs any PRD where a screen has fewer than 4 states (empty/loading/populated/error). Single-state screens are the most common failure mode.
- **Don't start a SaaS-connector product without OAuth-app registration as a parallel owner track.** Engineering can scaffold OAuth callbacks instantly; what blocks shipping is platform-side: Shopify Partner account (free, 30 min), Meta dev app + App Review (3–7 days for `ads_read`), Google Ads dev token (2–4 wk!), Razorpay Partner approval (1–3 days). Apply Day 0; ship key-paste path while you wait. *Learned on D2C OS V1.*
- **OAuth app registration is one-time per platform, not per-merchant.** d2cos registers as an "app" with each platform once → every merchant who ever signs up uses that single registered app. Owners often confuse "do I need a partnership for each connector" with "do I need a partnership for each customer" — the answer is one app, infinite customers.
- **Anthropic JWT-style anon keys + Supabase pooler IPv6 quirk + Shiprocket TLD split** — three independent landmines that cost time on D2C OS V1 setup. All three now in Decisions; bundle into a "Supabase + connector setup checklist" if we build another such product.
- **Indian SaaS auth fragility:** Many Indian SaaS APIs (Shiprocket, Interakt, AiSensy) don't offer OAuth — only API key/token. 2FA-enabled merchant accounts often can't use email+password login flows. Always offer a token-paste fallback path on Indian-platform connectors. *Reason: 2FA is becoming default for serious D2C brands; password-flow connectors will silently fail for them.*
- **Datadrew (datadrew.io) is the direct competitor for any "AI analyst for D2C" build:** Indian-founded, bootstrapped, 1000+ Shopify brands, $1.5B GMV analyzed, 5.0★ on Shopify App Store. Their gap = post-checkout (no Shiprocket/Razorpay/WhatsApp/Tally), pre-checkout-only. Differentiate on India-native ops connectors, not on chat-on-data architecture (they shipped that already).
- **The 14-day sprint slip risk is HIGH.** 10 agents in 2 weeks is heroic. Mitigation that survives advisor review: (a) Day 2 builds the runtime — not an agent; (b) every agent &lt; 200 LoC subclass on top of that runtime; (c) cut all V1.5+ "while you're there" features ruthlessly; (d) floor scenario = 5 agents documented as a hard floor. Without this discipline, sprint slips to 21+ days and we lose the Kwik AI race.
- **Don't dress V1.5+ internal tools as a Day-1 moat.** Sonnet advisor caught this. V1's actual moat = (1) Shiprocket connector layer (12+ engineer-years saved) + (2) 100K-merchant captive distribution at ₹0 CAC. The internal-tool stack (GEO, Email engine, Landing builder, etc) is a 24-month compounding story — frame as "where the moat compounds over 24 months", not as if Ring 2 tools exist on Day 1.
- **Every ₹ outcome claim must be derivable inline.** "₹3-5L/mo recovered" is not a number, it's a vibe. The defensible form: "₹50L GMV × 60% COD share × 30% RTO ≈ ₹9L failed/mo. Recover 40-55% via WA confirm = ₹3.6-5L/mo." Sonnet advisor flagged un-sourced claims as the #1 CEO-trust killer. Footnote sources (Unicommerce 2026 RTO data, Shiprocket cart abandonment, Eshopbox weight-dispute industry data, aimnlaunch CAC trends) on first use.
- **TAM tables need source + SAM/SOM honesty tag.** A theoretical-ceiling TAM at 100% penetration discounts every other number on the page. Always add: "Theoretical at 100% penetration. Realistic SAM at year 5 = ~10-15% = ₹X." Source brand counts (IBEF D2C, Tracxn, internal panel data) explicitly.
- **The multi-product chain (5-tool decision) is the #1 differentiator vs Kwik AI.** Kwik AI is locked to GoKwik checkout — they can't see Shopify or Razorpay outside their own stack. Triple Whale doesn't act on Interakt. Custom integrations cost ₹10L+ each. The "Summer-Hero campaign rescue" worked example (read Meta → join Shopify customers + Shiprocket RTO + Razorpay payment status → act on Engage WA + queue Creative Co-pilot) stays in both Strategy and PRD as the headline capability. Never cut.
- **Backend deploy on Railway is blocked at free tier.** User has 2 existing projects (visibl, angelic-acceptance) maxing the cap. Either (a) cleanup one (visibl is paused per brain), (b) upgrade to Hobby ($5/mo) — recommended. Don't try `railway init` again until slot freed.

## Preferences
*How the project owner wants things done. These override defaults.*

- Naming suggestions for this project should be personal — TG / Tanish-inspired

- **Final stakeholder PDFs/files in `outputs/<slug>/prd/` named as `[Brand]-[Type]-vN.pdf`.** Source HTML next to PDF for editing. Build directory `build/` for assembly artifacts (head template, body fragments) — gitignored.
- **rocketizer.ai is the brand-spelling reference.** "Rocketizer" with z, "Atlas" tier, Twenty2 voice product, 5 pillars (F.01–F.05), 25+ named agents catalogue, ₹49K/₹1.5L/custom pricing tiers — match these conventions when writing for stakeholders.

Level 2 = real engineering (code sketches, file structure, the actual loop). Level 3 = where it breaks in production (failure modes, security cliffs, vocabulary fixes). End deep-dives by offering the next layer or actionable artifact — let owner pick.

Don't soften. The value-add of these sessions is converting a 80%-right model to a 100%-right one — not validating the 80%.
